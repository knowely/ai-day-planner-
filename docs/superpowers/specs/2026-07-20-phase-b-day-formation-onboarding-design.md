# AI-планер дня — Фаза B: авто-формування Today, онбординг, порожні екрани

**Дата:** 2026-07-20
**Статус:** затверджено на етапі брейнштормінгу, готово до плану реалізації

Це Фаза B, описана в `docs/superpowers/specs/2026-07-20-ai-task-parsing-design.md` як наступний крок після Фази A. Складається з трьох незалежних, але пов'язаних частин:
- **B1:** AI сам формує список Today з беклогу Inbox (замість лише ручного «→ Сьогодні»).
- **B2:** Вступний екран, що показується один раз новому користувачу.
- **B3:** Змістовні порожні екрани на Capture/Inbox/Today.

## B1. AI формує Today

### Модель і провайдер

Той самий `OPENROUTER_API_KEY` (вже налаштований локально й на Vercel) і та сама модель `anthropic/claude-haiku-4.5`, що й у `src/app/api/parse-tasks/route.ts`. Новий роут **дзеркалить** усі технічні рішення цього файлу: структура tool-calling запиту, обробка помилок (400/500/502), `AbortSignal.timeout`, ручна валідація тіла запиту без бібліотек.

**Чому tool calling, а не вільний JSON-текст** (відхилення від початкової ідеї "розпарсь відповідь безпечно try/catch"): проєкт уже вирішив цю дилему в Фазі A на користь tool calling саме тому, що це гарантує структуру на рівні API, а не лише промптом. Новий роут іде тим самим шляхом для консистентності й надійності.

### `src/app/api/plan-day/route.ts`

**Вхід:** `{ backlog: Array<{ id: string; text: string; priority: TaskPriority; estimatedMinutes: number | null; deadline: string | null }> }`

**Вихід (успіх):** `200 { taskIds: string[] }` — id вибраних задач у порядку виконання.
**Вихід (помилка):** `4xx/5xx { error: string }`:
- порожній/відсутній/невалідний `backlog` (не масив, або масив без елементів) → `400`
- немає `OPENROUTER_API_KEY` → `500`
- мережева помилка / timeout до OpenRouter → `502`
- OpenRouter повернув не-200 → `502`
- модель не викликала tool `plan_day` або аргументи не парсяться як JSON → `502`

**Промпт (система):** передає сьогоднішню дату (як і в `parse-tasks`) та критерії відбору: пріоритет (вищий — важливіший), дедлайн (ближчий — важливіший), сумарний орієнтовний час — не перевантажувати день (орієнтир ~6 год / 360 хв активних задач; де `estimatedMinutes` невідомий — діяти на власний розсуд моделі). Модель бачить лише `id`/`text`/`priority`/`estimatedMinutes`/`deadline` кожної задачі беклогу — без зайвих полів.

**Tool-схема:**
```json
{
  "type": "function",
  "function": {
    "name": "plan_day",
    "description": "Select and order backlog tasks that should be done today, respecting priority, deadline urgency, and a ~6-hour total time budget.",
    "parameters": {
      "type": "object",
      "properties": {
        "taskIds": {
          "type": "array",
          "items": { "type": "string" },
          "description": "IDs of selected backlog tasks, in the order they should be tackled today."
        }
      },
      "required": ["taskIds"]
    }
  }
}
```

**Валідація відповіді** (нова функція `sanitizePlanDayResponse(raw: unknown, validIds: Set<string>): string[]` у `src/lib/planDayResponse.ts`, за зразком `parseTasksResponse.ts`): якщо `raw.taskIds` не масив рядків → `[]`; кожен id, якого немає серед `validIds` (реальний беклог, надісланий у запиті) — відкидається; дублікати — відкидаються (лишається перше входження); порядок AI — зберігається. Роут повертає `{ taskIds: sanitizePlanDayResponse(...) }`.

### Клієнт: `useTasks()` — новий мутатор

`applyDayPlan(orderedIds: string[]): void` — для кожного `id` (у наданому порядку) знаходить відповідну задачу зі `status: 'inbox'`, переводить у `status: 'today'`, і розміщує ці задачі **на початку** масиву (перед усіма вже наявними — ручними чи з попереднього AI-формування). Id, яких немає в поточному беклозі (захист і на клієнті), ігноруються. Порожній результуючий вибір → без змін стану.

### Клієнт: Today-екран

- Кнопка **«✨ Сформувати день»** видима завжди, коли є хоч одна задача зі `status: 'inbox'` (беклог), незалежно від того, чи Today вже щось містить — це дозволяє пересформувати план.
- Клік: `POST /api/plan-day` з поточним беклогом → показ стану завантаження **«AI планує твій день…»**, кнопка задизейблена → успіх: `applyDayPlan(taskIds)` → невдача (мережа/timeout/не-200/невалідна відповідь): показати повідомлення про помилку під кнопкою, **стан задач не змінюється** (жодного тихого fallback — для вибору "що робити сьогодні" немає безпечного дефолтного значення, на відміну від розбиття тексту по рядках у Фазі A).
- Ручна кнопка «→ Сьогодні» в Inbox лишається без змін.

## B2. Вступний екран (один раз)

### Компонент `src/components/OnboardingOverlay.tsx`

Повноекранний оверлей, що рендериться всередині Capture-екрана (`src/app/page.tsx`), не в `layout.tsx` — це свідомо: онбординг завжди веде саме на Capture, а раз компонент і так змонтований на Capture, не потрібна навігація між сторінками чи міжкомпонентний сигнал для фокусу поля.

**Прапорець:** `localStorage` ключ `ai-day-planner:onboarding-seen` (той самий префікс неймспейсу, що й `ai-day-planner:tasks`). Той самий анти-hydration-mismatch патерн, що й у `useTasks`: стан стартує як `false` (щоб збігтися з серверним рендером), перевірка `localStorage` — тільки в `useEffect` після монтування.

**Вміст:**
- Заголовок: «Плануй день голосом»
- Підзаголовок: «Надиктуй усе, що в голові. AI розкладе це на задачі — з пріоритетом, часом і дедлайном — і сам складе твій план на сьогодні.»
- Три квадратні картки в сітці 3 колонки (`grid grid-cols-3 gap-3`, кожна `aspect-square`): ✏️ Capture / 📥 Inbox / ✅ Today — з коротким підписом під іконкою, ті самі іконки, що вже використовує `BottomNav`.
- Кнопка **«Почати»**: ставить прапорець `true` в `localStorage`, ховає оверлей (локальний state в Capture-екрані), одразу фокусує textarea (простий `ref.current?.focus()`, оверлей і поле — в одному дереві, навігація не потрібна).

## B3. Розумні порожні екрани

### Capture (`src/app/page.tsx`)

Під textarea, **тільки коли вона порожня і немає активного `displayMessage`** (стан запису/розпізнавання/помилки мікрофона має пріоритет над підказкою):
```
Натисни 🎤 і просто проговори все, що треба зробити.
Напр.: «Завтра прибрати квартиру, це важливо, десь година. І зібрати валізу.»
```
Стиль — той самий приглушений сірий (`text-zinc-500 dark:text-zinc-400`), що й існуючі повідомлення.

### Inbox (`src/app/inbox/page.tsx`)

Коли `inboxTasks.length === 0`: іконка 📥, текст «Inbox поки порожній. Тут з'являться задачі, щойно ти щось надиктуєш.», кнопка **«← У Capture»** — звичайний `<Link href="/">` (та сама навігаційна модель, що й у `BottomNav`).

### Today (`src/app/today/page.tsx`)

Кнопка «✨ Сформувати день» (з B1) рендериться завжди, коли `backlogCount > 0` (де `backlogCount = tasks.filter(t => t.status === 'inbox').length`), **над** списком/порожнім станом. Порожній стан Today (`todayTasks.length === 0`) залежить від `backlogCount`:
- `backlogCount === 0`: «Спершу додай задачі в Inbox — і AI складе твій день.» (кнопки немає — формувати нема з чого)
- `backlogCount > 0`: «У беклозі {backlogCount} задач.» (кнопка вже показана вище, тут дублювання дії не потрібне)

## Тестування

- **`src/lib/planDayResponse.ts`**: валідні id проходять у заданому порядку; невідомі id відкидаються; дублікати відкидаються (лишається перше входження); не-масив/відсутнє поле → `[]`.
- **`src/app/api/plan-day/route.ts`**: успішний виклик з валідацією проти беклогу; порожній/невалідний `backlog` → 400; відсутній ключ → 500; мережева помилка/timeout → 502; не-200 від OpenRouter → 502; модель не викликала tool → 502; невалідний JSON у tool-аргументах → 502.
- **`useTasks().applyDayPlan`**: переносить задачі з inbox у today, розміщує їх на початку масиву перед наявними; ігнорує id поза беклогом; порожній вибір — без змін стану.
- **`OnboardingOverlay`**: показується, коли прапорця немає; не показується, коли прапорець уже `true`; клік «Почати» ставить прапорець і викликає `onStart`.
- **Today-екран**: кнопка формування з'являється/зникає залежно від `backlogCount`; стан завантаження; повідомлення про помилку без зміни задач; правильний порожній текст для обох підвипадків.
- **Inbox/Capture порожні екрани**: правильний текст/кнопка рендериться за відповідних умов.

## Ручний чек-лист перевірки

1. Очистити `localStorage` → відкрити застосунок → бачити вступний екран з трьома картками; тиснути «Почати» → екран ховається, курсор одразу в полі Capture.
2. Перезавантажити сторінку → вступний екран більше не з'являється.
3. З порожнім Inbox і порожнім Today → на Capture бачити підказку з прикладом; на Inbox — «Inbox поки порожній…» з кнопкою «← У Capture»; на Today — «Спершу додай задачі в Inbox…».
4. Надиктувати/ввести кілька задач різного пріоритету → в Today побачити повідомлення «У беклозі N задач.» і кнопку «✨ Сформувати день».
5. Натиснути «✨ Сформувати день» → бачити «AI планує твій день…» → задачі з'являються в Today, у розумному порядку (пріоритет/дедлайн), сумарний час не зашкалює за ~6 год.
6. Вручну перенести ще одну задачу через «→ Сьогодні» в Inbox, потім ще раз натиснути «✨ Сформувати день» → нові AI-задачі додаються зверху, вручну перенесена лишається на місці, нічого не губиться.
7. Вимкнути мережу → натиснути «✨ Сформувати день» → бачити повідомлення про помилку, список Today не змінюється.
