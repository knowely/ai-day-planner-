# AI-планер дня — AI-розбиття тексту на задачі (Фаза 2)

**Дата:** 2026-07-20
**Статус:** затверджено на етапі брейнштормінгу, готово до плану реалізації

## 1. Суть

Один наскрізний сценарій: хаотичний текст із Capture-екрана → структурований масив задач через AI, замість поточного примітивного розбиття по рядках. Це MVP Фази 2 з дорожньої карти — нічого понад цей сценарій (без авто-розкладки по днях, без пріоритетів/дедлайнів — це поза межами й цієї фази).

## 2. Провайдер і модель

- **OpenRouter**, не пряме звернення до Anthropic API — ключ уже збережено у Vercel як `OPENROUTER_API_KEY`.
- Ендпоінт: `https://openrouter.ai/api/v1/chat/completions` (OpenAI-сумісний формат).
- Модель: `anthropic/claude-haiku-4.5`.
- Структурований вивід — через **tool calling**, а не парсинг вільного тексту: модель змушена викликати функцію `extract_tasks` зі схемою `{ tasks: string[] }`. Це і є реалізація принципу «не довіряй відповіді моделі на слово» — структура гарантована API, а не тільки промптом.

## 3. Що робить AI з текстом

Розбиває потік думок на окремі задачі (навіть якщо кілька задач в одному рядку чи одна думка розбита на кілька рядків) **і легко очищує формулювання**: прибирає слова-паразити («ага», «емм»), явні друкарські помилки голосового вводу, починає кожну задачу з великої літери. Без перефразування змісту, без додавання деталей, яких не було в оригіналі.

## 4. Архітектура

### Сервер: `src/app/api/parse-tasks/route.ts`

Next.js Route Handler, `POST`.

**Вхід:** `{ text: string }`
**Вихід (успіх):** `200 { tasks: string[] }`
**Вихід (помилка):** `4xx/5xx { error: string }` — у таких випадках:
- порожній/відсутній `text` у тілі запиту → `400`
- немає `OPENROUTER_API_KEY` у середовищі → `500` (конфігураційна помилка, не показувати деталі клієнту)
- OpenRouter повернув помилку (мережа, rate limit, немає credits) → `502`
- модель не викликала `extract_tasks` або повернула невалідний JSON/не-масив/масив не-рядків → `502` (трактується як «AI не впорався», не як успіх із сміттям)

Валідація відповіді моделі: `tasks` має бути масивом, кожен елемент — непорожній (після trim) рядок; обрізати до ліміту 50 задач за один запит (захист від зловживання/дуже довгого тексту); порожні/пробільні елементи відфільтровуються.

`OPENROUTER_API_KEY` читається виключно через `process.env` всередині цього серверного файлу — ніколи не потрапляє в клієнтський бандл.

### Клієнт: `useTasks()` — новий метод

`addParsedTasks(texts: string[]): void` — додає вже готовий масив рядків як `inbox`-задачі напряму (без повторного розбиття по рядках, на відміну від існуючого `addTasksFromText`, який лишається для fallback-сценарію). Порожні рядки в масиві ігноруються (той самий фільтр, що і в `parseCaptureText`).

### Клієнт: Capture-екран (`src/app/page.tsx`)

Кнопка «Додати» при кліку:
1. Якщо текст порожній (після trim) — нічого не робити (як і зараз).
2. Інакше: показати стан завантаження на кнопці (задизейблити, змінити текст/спінер), викликати `POST /api/parse-tasks` з `{ text }`.
3. **Успіх** (200, валідний `tasks`): викликати `addParsedTasks(tasks)`, очистити textarea, зняти стан завантаження.
4. **Невдача** (мережева помилка, не-200, невалідна відповідь, timeout): **мовчки fallback** на існуючу поведінку — `addTasksFromText(text)` (порядкове розбиття по рядках), очистити textarea, зняти стан завантаження. Користувач не бачить помилки й не втрачає текст — гіршому разі отримує той самий результат, що й зараз у Фазі 1.
5. Timeout на клієнтський запит: 15 секунд — якщо AI довго не відповідає, тригериться той самий fallback.

## 5. Обробка помилок (підсумок)

- Немає AI-ключа / AI недоступний / модель повернула сміття → тихий fallback на порядкове розбиття, застосунок ніколи не блокується і не падає.
- Занадто довгий вхідний текст або дуже багато задач у відповіді → обрізається лімітом на сервері (50 задач), не помилка.
- Мережевий timeout на клієнті → той самий fallback.

## 6. Тестування

- **Серверний route** (`src/app/api/parse-tasks/route.test.ts`): мокати `fetch` до OpenRouter — кейси: успішний tool call з валідним масивом, модель не викликала tool, tool-виклик з невалідним JSON, tool-виклик з масивом не-рядків, порожній `text` у запиті, відсутній `OPENROUTER_API_KEY`, мережева помилка від OpenRouter, більше 50 задач у відповіді (перевірити обрізання).
- **`useTasks.addParsedTasks`**: аналогічно існуючим тестам `addTasksFromText` — додає задачі, ігнорує порожні рядки в масиві.
- **Capture-екран**: мокати `fetch` — успішний AI-виклик додає задачі з відповіді API (не з локального розбиття); мережева помилка/невалідна відповідь викликає fallback на `addTasksFromText`; кнопка задизейблена під час запиту.

## 7. Ручний чек-лист перевірки

1. Ввести хаотичний текст кількома реченнями (включно з задачами в одному рядку) на Capture, натиснути «Додати» → в Inbox з'являються чисто розбиті окремі задачі, а не по одній на кожен вихідний рядок.
2. Продиктувати текст голосом (Web Speech API) із мовними паразитами/фрагментами → AI-розбиття прибирає сміття.
3. Вимкнути мережу (DevTools offline) → натиснути «Додати» → після паузи задачі все одно з'являються в Inbox (fallback на порядкове розбиття), застосунок не показує помилку й не втрачає текст.
4. Перевірити у Vercel, що після деплою з новим кодом `OPENROUTER_API_KEY` реально використовується (запит доходить до OpenRouter, credits списуються).
