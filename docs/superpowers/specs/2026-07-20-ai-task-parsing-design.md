# AI-планер дня — AI-розбиття тексту на задачі (Фаза 2, Фаза A)

**Дата:** 2026-07-20
**Статус:** затверджено на етапі брейнштормінгу, готово до плану реалізації

Це Фаза A ширшої Фази 2 (AI-планер у стилі Todoist). Повна візія Фази 2 складається з двох кроків:
- **Фаза A (цей документ):** AI розбиває хаотичний текст на задачі й визначає для кожної пріоритет/орієнтовний час/дедлайн. Перенесення з Inbox у Today лишається **ручним** (кнопка «→ Сьогодні»).
- **Фаза B (окремий майбутній спек):** AI сам формує список Today з беклогу Inbox, без ручного перенесення.

Розбито на дві фази свідомо — щоб не проєктувати логіку авто-формування Today (Фаза B) наосліп, поки немає реальних даних (пріоритет/дедлайн) з Фази A, на яких ця логіка мала б будуватись.

## 1. Суть

Один наскрізний сценарій: хаотичний текст із Capture-екрана → структурований масив задач через AI (текст + пріоритет + орієнтовний час + дедлайн), замість поточного примітивного розбиття по рядках без жодних атрибутів.

## 2. Провайдер і модель

- **OpenRouter**, не пряме звернення до Anthropic API — ключ уже збережено у Vercel як `OPENROUTER_API_KEY`.
- Ендпоінт: `https://openrouter.ai/api/v1/chat/completions` (OpenAI-сумісний формат).
- Модель: `anthropic/claude-haiku-4.5`.
- Структурований вивід — через **tool calling**, а не парсинг вільного тексту: модель змушена викликати функцію `extract_tasks` зі строгою JSON-схемою. Це реалізація принципу «не довіряй відповіді моделі на слово» — структура гарантована API, а не тільки промптом; сервер додатково валідує кожне поле (розділ 4).

## 3. Що робить AI з текстом

Для кожної задачі, розпізнаної в тексті:
- **Розбиває** потік думок на окремі задачі (навіть якщо кілька задач в одному рядку чи одна думка розбита на кілька рядків) і **легко очищує формулювання**: прибирає слова-паразити («ага», «емм»), явні друкарські помилки голосового вводу, починає з великої літери. Без перефразування змісту.
- **Визначає пріоритет** — `low` / `medium` / `high`, на основі тону й слів у тексті («терміново», «важливо», «колись» тощо). Якщо сигналів немає — `medium`.
- **Оцінює орієнтовну тривалість** (`estimatedMinutes`) — число хвилин, якщо з тексту/здорового глузду можна оцінити («на 15 хв», «швидкий дзвінок» → мало часу, «зробити звіт» → більше). `null`, якщо оцінити нема з чого.
- **Визначає дедлайн** — AI сам оцінює терміновість навіть без явної вказівки в тексті (не тільки коли прямо написано «до п'ятниці», а й неявно, з контексту). Модель отримує сьогоднішню дату в промпті, щоб «завтра»/«до п'ятниці» перетворювались у конкретну дату. `null`, якщо дедлайну справді немає (задача не термінова і в тексті/контексті нема на це натяку).

## 4. Архітектура

### Модель задачі — нові поля

```ts
interface Task {
  id: string
  text: string
  status: 'inbox' | 'today'
  done: boolean
  createdAt: number
  priority: 'low' | 'medium' | 'high'
  estimatedMinutes: number | null
  deadline: string | null // ISO-дата YYYY-MM-DD
}
```

### Сервер: `src/app/api/parse-tasks/route.ts`

Next.js Route Handler, `POST`.

**Вхід:** `{ text: string }`
**Вихід (успіх):** `200 { tasks: ParsedTask[] }`, де
```ts
interface ParsedTask {
  text: string
  priority: 'low' | 'medium' | 'high'
  estimatedMinutes: number | null
  deadline: string | null
}
```
**Вихід (помилка):** `4xx/5xx { error: string }` — у таких випадках:
- порожній/відсутній `text` у тілі запиту → `400`
- немає `OPENROUTER_API_KEY` у середовищі → `500` (конфігураційна помилка, деталі клієнту не показуються)
- OpenRouter повернув помилку (мережа, rate limit, немає credits) → `502`
- модель не викликала `extract_tasks` або повернула невалідний JSON/не-масив → `502` (трактується як «AI не впорався», не як успіх зі сміттям)

**Валідація кожного елемента відповіді моделі** (нічому з відповіді AI не довіряємо без перевірки):
- `text` — має бути непорожнім (після trim) рядком; елементи з порожнім `text` відкидаються
- `priority` — якщо не один із `'low' | 'medium' | 'high'` (відсутнє, не той тип, невідоме значення) → підставляється `'medium'`
- `estimatedMinutes` — `null`, або число; від'ємне чи нечислове → `null`; більше 480 → обрізається до 480
- `deadline` — `null`, або рядок у форматі `YYYY-MM-DD`; будь-що, що не проходить парсинг як валідна дата в цьому форматі → `null`
- масив `tasks` обрізається до 50 елементів (захист від зловживання/дуже довгого тексту)

Сервер передає моделі поточну дату (`new Date().toISOString().slice(0, 10)`) в системному промпті — потрібно для коректного розрахунку відносних дедлайнів.

`OPENROUTER_API_KEY` читається виключно через `process.env` всередині цього серверного файлу — ніколи не потрапляє в клієнтський бандл.

### Клієнт: `src/lib/tasks.ts` — оновлення

`createTask` (та відповідний фабричний шлях для AI-задач) генерує `Task` із новими полями. Для fallback-шляху (порядкове розбиття по рядках, без AI) нові поля виставляються в дефолти: `priority: 'medium'`, `estimatedMinutes: null`, `deadline: null`.

### Клієнт: `useTasks()` — новий метод

`addParsedTasks(parsed: ParsedTask[]): void` — перетворює кожен `ParsedTask` на повний `Task` (генерує `id`/`createdAt`/`status: 'inbox'`/`done: false`, переносить `text`/`priority`/`estimatedMinutes`/`deadline`) і додає в стан. Елементи з порожнім `text` ігноруються (той самий фільтр духу, що і в `parseCaptureText`).

### Клієнт: Capture-екран (`src/app/page.tsx`)

Кнопка «Додати» при кліку:
1. Якщо текст порожній (після trim) — нічого не робити (як і зараз).
2. Інакше: показати стан завантаження на кнопці (задизейблити, змінити текст/спінер), викликати `POST /api/parse-tasks` з `{ text }`.
3. **Успіх** (200, валідний `tasks`): викликати `addParsedTasks(tasks)`, очистити textarea, зняти стан завантаження.
4. **Невдача** (мережева помилка, не-200, невалідна відповідь, timeout): **мовчки fallback** на існуючу поведінку — `addTasksFromText(text)` (порядкове розбиття по рядках, задачі з дефолтними `priority: 'medium'`, `estimatedMinutes: null`, `deadline: null`), очистити textarea, зняти стан завантаження. Користувач не бачить помилки й не втрачає текст.
5. Timeout на клієнтський запит: 15 секунд — якщо AI довго не відповідає, тригериться той самий fallback.

### Клієнт: Inbox/Today екрани — відображення нових полів

Під текстом кожної задачі — короткий рядок метаданих, частини розділені « · », показуються лише ті, що присутні (без плейсхолдерів на відсутні поля):
- кольорова крапка-пріоритет: 🔴 `high`, 🟡 `medium`, 🟢 `low`
- `~{estimatedMinutes} хв`, якщо не `null`
- дата дедлайну у форматі `дд.мм`, якщо не `null`

Якщо жодного поля немає (теоретично неможливо, бо `priority` завжди заповнене — крапка показується завжди; `estimatedMinutes`/`deadline` необов'язкові).

## 5. Обробка помилок (підсумок)

- Немає AI-ключа / AI недоступний / модель повернула сміття → тихий fallback на порядкове розбиття з дефолтними метаданими, застосунок ніколи не блокується і не падає.
- Невалідне значення окремого поля в відповіді AI (поганий `priority`/`estimatedMinutes`/`deadline`) → підставляється безпечний дефолт для цього поля, а не відкидається вся задача.
- Занадто довгий вхідний текст або дуже багато задач у відповіді → обрізається лімітом на сервері (50 задач), не помилка.
- Мережевий timeout на клієнті → той самий fallback.

## 6. Тестування

- **Серверний route** (`src/app/api/parse-tasks/route.test.ts`): мокати `fetch` до OpenRouter — кейси: успішний tool call з валідним масивом (усі поля коректні); модель не викликала tool; tool-виклик з невалідним JSON; елемент з порожнім `text` (відкидається); елемент з невідомим `priority` (замінюється на `medium`); елемент з від'ємним/нечисловим `estimatedMinutes` (замінюється на `null`); елемент з `estimatedMinutes` > 480 (обрізається до 480); елемент з невалідним форматом `deadline` (замінюється на `null`); порожній `text` у вхідному запиті; відсутній `OPENROUTER_API_KEY`; мережева помилка від OpenRouter; більше 50 задач у відповіді (перевірити обрізання).
- **`useTasks.addParsedTasks`**: додає задачі з усіма полями з `ParsedTask`, ігнорує елементи з порожнім `text`.
- **`addTasksFromText` (fallback)**: перевірити, що нові задачі отримують дефолти (`priority: 'medium'`, `estimatedMinutes: null`, `deadline: null`).
- **Capture-екран**: мокати `fetch` — успішний AI-виклик додає задачі з відповіді API; мережева помилка/невалідна відповідь викликає fallback; кнопка задизейблена під час запиту.
- **Inbox/Today екрани**: рядок метаданих рендериться коректно для комбінацій присутніх/відсутніх `estimatedMinutes`/`deadline`; крапка-пріоритет відповідає значенню.

## 7. Ручний чек-лист перевірки

1. Ввести хаотичний текст кількома реченнями з різною терміновістю («терміново подзвонити в банк», «колись помити вікна») → в Inbox задачі мають різний пріоритет і, де доречно, дедлайн.
2. Продиктувати текст голосом (Web Speech API) із мовними паразитами → AI-розбиття прибирає сміття, метадані все одно проставляються.
3. Вимкнути мережу (DevTools offline) → натиснути «Додати» → після паузи задачі все одно з'являються в Inbox (fallback, без пріоритету/дедлайну — просто `medium` і без дедлайну), застосунок не показує помилку.
4. Перевірити на екрані телефону, що рядок метаданих під задачею не ламає верстку й не заважає тиснути кнопки.
5. Перевірити у Vercel, що після деплою `OPENROUTER_API_KEY` реально використовується (запит доходить до OpenRouter, credits списуються).
